# SCOPE.md — Data Anomaly Log & Database Schema

## Overview

This document records every data anomaly encountered during development and data migration for the SplitWise Clone project, and describes how each was handled. It also contains the full database schema.

---

## Part 1: Data Anomaly Log

### Source: SQLite → Neon PostgreSQL Migration (`migrate_data.py`)

The project initially used a local SQLite database for development, then migrated all data to a Neon PostgreSQL instance for production.

---

### Anomaly 1 — Django `post_save` Signal Conflict on User Import

| Field | Detail |
|---|---|
| **Problem** | When loading the SQLite fixture dump into PostgreSQL, Django's `post_save` signal (`create_user_settings`) auto-fired for every User record being re-imported, attempting to create duplicate `UserProfile`, `NotificationPreference`, and `PrivacySettings` rows. This caused `UNIQUE constraint failed` database errors. |
| **Detection** | The `loaddata` command raised `IntegrityError: duplicate key value violates unique constraint "splitwise_api_userprofile_user_id_key"` for every user row after the first. |
| **Action Taken** | Disconnected the `post_save` signal before running `loaddata`, then reconnected it after. This is the standard Django pattern for bulk imports where related objects already exist in the fixture. |
| **Code Reference** | `migrate_data.py`, lines 54–62 |

```python
post_save.disconnect(create_user_settings, sender=User)
call_command('loaddata', dump_file)
```

---

### Anomaly 2 — Foreign Key Natural Key Resolution Failure

| Field | Detail |
|---|---|
| **Problem** | Default Django fixture exports use integer primary keys for foreign key references. When migrating across databases, auto-increment IDs in SQLite and PostgreSQL do not align, causing FK integrity errors on import. |
| **Detection** | Initial `dumpdata` without `--natural-foreign` produced fixture with numeric IDs. When loaded into PostgreSQL (which had different auto-increment sequences), FK constraints failed for `Expense → Group`, `ExpenseSplit → Expense`, `Settlement → Group`. |
| **Action Taken** | Added `--natural-foreign --natural-primary` flags to `dumpdata`. Django then uses semantic identifiers (e.g., `username` for User) instead of integer PKs, making the fixture database-agnostic. |
| **Code Reference** | `migrate_data.py`, line 36 |

```bash
dumpdata --natural-foreign --natural-primary -e contenttypes -e auth.Permission -e sessions.Session
```

---

### Anomaly 3 — `.env` File Collision During Migration

| Field | Detail |
|---|---|
| **Problem** | During migration, the `.env` file (pointing to Neon PostgreSQL) needed to be temporarily hidden so Django's `dumpdata` would connect to the local SQLite database. Running `dumpdata` with the PostgreSQL `.env` active would dump production data instead of local test data. |
| **Detection** | First attempt at migration accidentally dumped an empty database (the blank Neon PostgreSQL instance) instead of the populated local SQLite. |
| **Action Taken** | `migrate_data.py` renames `.env` to `.env.bak` before dumping, then restores it before loading, ensuring SQLite is used for export and PostgreSQL for import. A `try/finally` block guarantees `.env` is always restored even if the dump step fails. |
| **Code Reference** | `migrate_data.py`, lines 22–45 |

---

### Anomaly 4 — `contenttypes` and `auth.Permission` Conflicts

| Field | Detail |
|---|---|
| **Problem** | Including Django's built-in `contenttypes` and `auth.Permission` tables in the fixture caused import failures because PostgreSQL already had these auto-generated by `migrate`. Importing duplicate rows violated unique constraints. |
| **Detection** | Error: `IntegrityError: duplicate key value violates unique constraint "django_content_type_app_label_model_key"` |
| **Action Taken** | Excluded `contenttypes`, `auth.Permission`, and `sessions.Session` from `dumpdata` using the `-e` flag. These tables are auto-managed by Django and should never be manually imported. |

---

### Anomaly 5 — Decimal Precision: `split_value` Nullable Mismatch

| Field | Detail |
|---|---|
| **Problem** | For `equal` split type, no `split_value` is needed. SQLite stored `None` (NULL) for these rows. When PostgreSQL validated this, the field had to be explicitly defined as `null=True, blank=True` or the migration would fail with NOT NULL constraint errors. |
| **Detection** | `IntegrityError: null value in column "split_value" violates not-null constraint` during first migration attempt. |
| **Action Taken** | Added `null=True, blank=True` to `ExpenseSplit.split_value` field. For equal splits, `split_value` is intentionally NULL; for percentage/shares/unequal splits, it holds the relevant numeric value. |

---

### Anomaly 6 — Avatar Storage: File Path vs Base64

| Field | Detail |
|---|---|
| **Problem** | The original design used Django's `ImageField` with file-path storage. On Render's ephemeral filesystem, uploaded files are lost on each deployment. |
| **Detection** | After deploying to Render, uploaded profile photos disappeared after every redeploy since Render does not have persistent disk storage on the free tier. |
| **Action Taken** | Changed `UserProfile.avatar` from `ImageField` to `TextField` storing Base64-encoded image data directly in PostgreSQL. This eliminates filesystem dependency entirely. Comment in code: `# Store Base64 string directly for Render compatibility`. |

---

### Anomaly 7 — JWT `jti` Uniqueness Across Database Reset

| Field | Detail |
|---|---|
| **Problem** | `UserSession.jti` (JWT token identifier) must be globally unique. After importing fixture data into a fresh PostgreSQL instance, any stale JWTs in clients pointed to `jti` values that no longer existed in `UserSession`, causing silent auth failures. |
| **Detection** | Existing browser sessions with valid JWT tokens received 401 errors after migration because `UserSession` rows were not in the new database. |
| **Action Taken** | Cleared `localStorage` tokens after database migration. `UserSession` rows were included in the fixture export, but users were advised to log in again post-migration. |

## Part 2: Database Schema

### Entity Relationship Overview

```
User (Django built-in)
 ├── UserProfile (1:1)
 ├── NotificationPreference (1:1)
 ├── PrivacySettings (1:1)
 ├── UserSession (1:N)
 ├── BlockedUser (N:N self-referential via two FKs)
 ├── Expense (1:N via payer)
 ├── ExpenseSplit (1:N)
 ├── Settlement (1:N via payer and payee)
 └── ChatMessage (1:N)

Group
 ├── members (M:N → User)
 ├── Expense (1:N)
 └── Settlement (1:N)

Expense
 ├── group (FK → Group)
 ├── payer (FK → User)
 ├── ExpenseSplit (1:N)
 └── ChatMessage (1:N)
```
